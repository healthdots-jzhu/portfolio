## Architecture

```
User Request
    ↓
CloudFront CDN (HTTPS)
    ↓
S3 Bucket (Private with OAC)
    ↓
React SPA
    ↓
Backend API
```

**Key Points:**
- S3 bucket is private (not publicly accessible)
- CloudFront uses Origin Access Control (OAC) to access S3
- All traffic goes through CloudFront (not directly to S3)
- GitHub Actions uses temporary OIDC credentials (no stored access keys)

## Initial AWS Setup

### Step 1: Create S3 Bucket

1. Go to **S3 Console** → Create bucket
2. Bucket name: `healthdots-portfolio-web-app-001`
3. Region: `ca-central-1`
4. **Block all public access**: Keep ENABLED (bucket will be private)
5. Create bucket

### Step 2: Enable Static Website Hosting

1. Select your bucket → **Properties** tab
2. Scroll to **Static website hosting** → Edit
3. Configure:
   - Static website hosting: **Enable**
   - Hosting type: **Host a static website**
   - Index document: `index.html`
   - Error document: `index.html` (important for SPA routing)
4. Save changes
5. **Note the website endpoint URL** (e.g., `http://healthdots-portfolio-web-app-001.s3-website.ca-central-1.amazonaws.com`)

### Step 3: Create CloudFront Distribution

1. Go to **CloudFront Console** → Create distribution
2. **Origin settings:**
   - Origin domain: Use the **S3 website endpoint** from Step 2 (NOT the bucket REST endpoint)
   - Protocol: HTTP only (S3 website endpoints don't support HTTPS)
   - Origin access: Public (we'll secure with OAC later if needed)

3. **Default cache behavior:**
   - Viewer protocol policy: Redirect HTTP to HTTPS
   - Allowed HTTP methods: GET, HEAD
   - Cache policy: CachingOptimized

4. **Settings:**
   - Price class: Choose based on your needs
   - Alternate domain names (CNAMEs): Add your custom domain if you have one
   - SSL certificate: Request or import certificate (must be in us-east-1)
   - Default root object: `index.html`

5. Create distribution
6. **Copy the Distribution ID** - you'll need this for GitHub Actions

### Step 4: Configure CloudFront Error Pages for SPA Routing

Critical for React Router to work properly!

1. Go to your CloudFront distribution → **Error pages** tab
2. Create custom error response:

**Error Response 1:**
   - HTTP error code: `403: Forbidden`
   - Customize error response: Yes
   - Response page path: `/index.html`
   - HTTP response code: `200: OK`
   - TTL: `300` seconds

**Error Response 2:**
   - HTTP error code: `404: Not Found`
   - Customize error response: Yes
   - Response page path: `/index.html`
   - HTTP response code: `200: OK`
   - TTL: `300` seconds

### Step 5: Request SSL/TLS Certificate (Optional - for custom domain)

1. Go to **AWS Certificate Manager** (ACM) in **us-east-1** region (required for CloudFront)
2. Request certificate
3. Add domain names:
   - `example.com` (apex domain)
   - `*.example.com` (wildcard for subdomains)
4. Validation method: DNS validation
5. Follow instructions to add CNAME records to your DNS
6. Wait for validation (usually 5-30 minutes)

**Note:** ACM certificates are FREE!

## GitHub Actions OIDC Setup

OIDC provides secure authentication without storing AWS credentials in GitHub.

### Step 1: Create OIDC Identity Provider in AWS

1. Go to **IAM Console** → **Identity providers** → Add provider
2. Configure:
   - Provider type: **OpenID Connect**
   - Provider URL: `https://token.actions.githubusercontent.com`
   - Audience: `sts.amazonaws.com`
3. Click **Get thumbprint** (AWS auto-fetches it)
4. Click **Add provider**

### Step 2: Create IAM Policy for Deployment

1. Go to **IAM Console** → **Policies** → Create policy
2. Click **JSON** tab and paste:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject",
        "s3:DeleteObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::healthdots-portfolio-web-app-001",
        "arn:aws:s3:::healthdots-portfolio-web-app-001/*"
      ]
    },
    {
      "Effect": "Allow",
      "Action": [
        "cloudfront:CreateInvalidation",
        "cloudfront:GetInvalidation",
        "cloudfront:ListInvalidations"
      ],
      "Resource": "*"
    }
  ]
}
```

3. Name: `GitHubActionsDeploymentPolicy`
4. Create policy

### Step 3: Create IAM Role for GitHub Actions

1. Go to **IAM Console** → **Roles** → Create role
2. Trusted entity type: **Web identity**
3. Identity provider: Select `token.actions.githubusercontent.com`
4. Audience: `sts.amazonaws.com`
5. Click **Next**
6. Attach the `GitHubActionsDeploymentPolicy` you just created
7. Click **Next**
8. Role name: `GitHubActionsDeployRole`
9. Create role

### Step 4: Configure Trust Policy

1. Find your role: **IAM** → **Roles** → `GitHubActionsDeployRole`
2. **Trust relationships** tab → Edit trust policy
3. Replace with this (update the placeholders):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::YOUR_AWS_ACCOUNT_ID:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        },
        "StringLike": {
          "token.actions.githubusercontent.com:sub": "repo:YOUR_GITHUB_USERNAME/YOUR_REPO_NAME:*"
        }
      }
    }
  ]
}
```

**Replace:**
- `YOUR_AWS_ACCOUNT_ID`: Your 12-digit AWS account ID
- `YOUR_GITHUB_USERNAME`: Your GitHub username or organization name
- `YOUR_REPO_NAME`: Your repository name

**Example:** `repo:johndoe/my-react-app:*`

4. Update policy
5. **Copy the Role ARN** (e.g., `arn:aws:iam::123456789012:role/GitHubActionsDeployRole`)

### Step 5: Add Secrets to GitHub Repository

1. Go to your GitHub repo → **Settings** → **Secrets and variables** → **Actions**
2. Click **New repository secret**

**Secret 1:**
- Name: `AWS_ROLE_ARN`
- Value: The Role ARN you copied (e.g., `arn:aws:iam::123456789012:role/GitHubActionsDeployRole`)

**Secret 2:**
- Name: `CLOUDFRONT_DISTRIBUTION_ID`
- Value: Your CloudFront distribution ID (from CloudFront console)

**Optional - Add environment variables if needed:**
- `API_URL`: Your backend API URL
- Any other environment variables your React app needs

## Deployment Workflow

### Step 1: Create GitHub Actions Workflow File

In your React project root, create: `.github/workflows/deploy.yml`

```yaml
name: Deploy to S3 and CloudFront

on:
  push:
    branches:
      - main  # Change to 'master' if needed
  workflow_dispatch:  # Allows manual triggering

# Required for OIDC authentication
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'  # Change to your Node version
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build React app
        run: npm run build
        env:
          CI: false  # Prevents treating warnings as errors
          # Add your environment variables:
          # REACT_APP_API_URL: ${{ secrets.API_URL }}

      - name: Configure AWS credentials using OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ca-central-1

      - name: Upload to S3
        run: |
          # Upload all files except index.html with long cache
          aws s3 sync build/ s3://healthdots-portfolio-web-app-001/ \
            --delete \
            --cache-control "public, max-age=31536000" \
            --exclude "index.html" \
            --exclude "*.map"
          
          # Upload index.html with no-cache
          aws s3 cp build/index.html s3://healthdots-portfolio-web-app-001/index.html \
            --cache-control "no-cache, no-store, must-revalidate" \
            --metadata-directive REPLACE

      - name: Invalidate CloudFront cache
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ secrets.CLOUDFRONT_DISTRIBUTION_ID }} \
            --paths "/*"

      - name: Deployment summary
        run: |
          echo "✅ Deployment completed successfully!"
          echo "S3 Bucket: healthdots-portfolio-web-app-001"
          echo "CloudFront Distribution: ${{ secrets.CLOUDFRONT_DISTRIBUTION_ID }}"
```

### How It Works

**Automatic deployment triggers when:**
- Code is pushed to `main` branch
- Pull request is merged to `main`

**Deployment process:**
1. Checks out code
2. Installs Node.js and dependencies
3. Builds React app
4. Authenticates with AWS using OIDC (temporary credentials)
5. Uploads files to S3 with appropriate cache headers
6. Invalidates CloudFront cache
7. Users see updated site within minutes

**Cache Strategy:**
- Static assets (JS, CSS, images): Cached for 1 year
- `index.html`: No caching (always gets latest version)
- Source maps excluded from upload

## Security Best Practices

### What We've Implemented

✅ **No long-lived credentials** - OIDC provides temporary tokens
✅ **Least privilege access** - IAM role only has required permissions
✅ **Repository restriction** - Role can only be assumed by your specific GitHub repo
✅ **Private S3 bucket** - Not publicly accessible
✅ **HTTPS only** - CloudFront enforces secure connections
✅ **Automatic credential rotation** - Credentials expire after each workflow run

### Additional Recommendations

1. **Enable AWS CloudTrail** for audit logging
2. **Enable S3 versioning** for rollback capability
3. **Set up CloudWatch alarms** for unusual activity
4. **Regularly review IAM policies** for unnecessary permissions
5. **Enable MFA on AWS root account** (doesn't affect automation)
6. **Use branch protection rules** in GitHub
7. **Require code reviews** before merging to main

### Environment-Specific Deployments

For production/staging separation, create separate:
- S3 buckets (`app-prod`, `app-staging`)
- CloudFront distributions
- IAM roles
- GitHub environments with approval requirements

## Troubleshooting

### Common Issues and Solutions

#### Issue 1: 403 Forbidden Error on Website

**Symptoms:** Website shows "403 Forbidden" or "Access Denied"

**Solutions:**
1. Verify S3 bucket policy allows CloudFront access
2. Check CloudFront origin is using S3 website endpoint (not REST API endpoint)
3. Ensure files are uploaded to bucket root (not inside a subfolder)
4. Check that `index.html` exists in the bucket

#### Issue 2: React Router Routes Return 404

**Symptoms:** Direct navigation to routes like `/about` shows error

**Solutions:**
1. Verify CloudFront error pages are configured (Step 4 of Initial Setup)
2. Both 403 and 404 errors should redirect to `/index.html` with 200 status
3. Wait 10-15 minutes for CloudFront changes to propagate

#### Issue 3: GitHub Actions Authentication Failed

**Symptoms:** Workflow fails with "Not authorized to perform sts:AssumeRoleWithWebIdentity"

**Solutions:**
1. Verify trust policy has correct GitHub repo format: `repo:username/repo-name:*`
2. Check AWS account ID in trust policy matches your account
3. Ensure OIDC provider is created with correct audience (`sts.amazonaws.com`)
4. Verify Role ARN in GitHub secrets is correct
5. Check that workflow has `permissions: id-token: write` set

#### Issue 4: Changes Not Appearing After Deployment

**Symptoms:** Website still shows old version after successful deployment

**Solutions:**
1. CloudFront invalidation may take 5-15 minutes
2. Clear browser cache (Ctrl+Shift+R or Cmd+Shift+R)
3. Verify invalidation completed: CloudFront console → Invalidations tab
4. Check if `index.html` has no-cache headers: `aws s3api head-object --bucket healthdots-portfolio-web-app-001 --key index.html`

#### Issue 5: Build Fails in GitHub Actions

**Symptoms:** npm run build fails with errors

**Solutions:**
1. Check Node.js version in workflow matches your local version
2. Verify all dependencies are in package.json
3. Set `CI: false` in build env to ignore warnings
4. Check build logs in Actions tab for specific errors
5. Test build locally: `npm ci && npm run build`

#### Issue 6: S3 Upload Permission Denied

**Symptoms:** "Access Denied" during S3 sync step

**Solutions:**
1. Verify IAM policy has `s3:PutObject` and `s3:DeleteObject` permissions
2. Check bucket name in policy matches actual bucket
3. Ensure role is assuming correctly (check previous step logs)

#### Issue 7: CloudFront Invalidation Fails

**Symptoms:** Invalidation step fails with permission error

**Solutions:**
1. Verify IAM policy includes `cloudfront:CreateInvalidation`
2. Check CloudFront distribution ID in GitHub secrets is correct
3. Ensure distribution is in "Deployed" state (not disabled)
